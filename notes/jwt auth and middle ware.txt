==============================
JWT AUTHENTICATION NOTES
==============================

1) WHAT PROBLEM JWT SOLVES
--------------------------
HTTP is stateless.
That means the server does not remember previous requests.

After login, the server must have a way to identify the user
on every new request.

JWT allows identity to travel with each request.


2) WHAT IS JWT
--------------
JWT = JSON Web Token

It is a signed string that contains:
- Header
- Payload
- Signature

Structure:
HEADER.PAYLOAD.SIGNATURE

Example:
xxxxx.yyyyy.zzzzz

JWT is SIGNED, not encrypted.
Anyone can decode payload.
But it cannot be modified without breaking the signature.


3) LOGIN FLOW
-------------
Step 1:
Client sends:
POST /login
{
  "email": "...",
  "password": "..."
}

Step 2:
Server:
- Finds user in DB
- Compares password using bcrypt.compare()

Step 3:
If valid:
Server generates token:

jwt.sign(
  { id: user.id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: "1h" }
)

Step 4:
Server sends token back to client.

Server does NOT store the token anywhere.


4) WHAT IS INSIDE THE TOKEN
----------------------------
Payload (in this project):
{
  id: user.id,
  role: user.role
}

JWT automatically adds:
- iat (issued at)
- exp (expiration time)

So decoded token looks like:
{
  id: 3,
  role: "staff",
  iat: ...,
  exp: ...
}


5) WHERE TOKEN IS STORED
------------------------
Server does NOT store it.

Token is stored on the client side.

In this project:
- Postman manually stores it.

In real apps:
- localStorage
- memory
- HTTP-only cookie

The token is attached to each request manually.


6) HOW TOKEN IS SENT
--------------------
Client sends protected request with header:

Authorization: Bearer <token>

The header is NOT storage.
It is only transport for that request.


7) AUTHENTICATION MIDDLEWARE
------------------------------
Middleware function:

function authenticateToken(req, res, next) {
  const authHeader = req.headers["authorization"];

  if (!authHeader) {
    return res.status(401).json({ error: "Access denied" });
  }

  const token = authHeader.split(" ")[1];

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded;
    next();
  } catch (err) {
    return res.status(403).json({ error: "Invalid or expired token" });
  }
}

What it does:
- Reads token from header
- Verifies signature using secret
- Extracts payload
- Attaches decoded payload to req.user
- Calls next() to continue request

If invalid:
- Blocks request


8) ROUTE PROTECTION
-------------------
To protect a route:

app.get("/protected", authenticateToken, handler)

Execution order:
1. authenticateToken runs
2. If valid -> next()
3. Route handler runs
4. If invalid -> blocked


9) AUTHENTICATION VS AUTHORIZATION
-----------------------------------

Authentication:
"Who are you?"
Handled by:
- Password verification
- JWT verification

Authorization:
"What are you allowed to do?"
Handled by:
- Role checks
- Ownership checks

Example:

if (req.user.role !== "admin") {
  return res.status(403).json({ error: "Forbidden" });
}


10) WHAT IS INSIDE req.user
---------------------------
req.user contains decoded payload:

{
  id: 3,
  role: "staff",
  iat: ...,
  exp: ...
}

This allows:
- Role-based access
- Ownership filtering
- No DB lookup for identity each request


11) SECURITY NOTES
------------------
- JWT secret must never be exposed.
- Do not store password in token.
- Tokens should expire (expiresIn).
- If token is stolen, it works until expiration.
- Role changes in DB do NOT update existing tokens.
- JWT is stateless (no session table).
